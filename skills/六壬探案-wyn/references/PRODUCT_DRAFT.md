# 大六壬·故事编织器 — 产品构思文档

> 状态：V2 讨论稿（已吸收优化建议）
> 更新：2026-03-27 14:29

---

## 一、产品定位

**一句话**：用大六壬术数模型驱动创意故事生成的交互式工作台。

**核心用户**：
- 创作者（编剧、小说作者）需要灵感推演工具
- 对传统文化/术数好奇但不懂大六壬的普通用户
- 想用"命运推演"框架来玩故事沙盘的人

**核心价值**：不懂六壬的人也能用六壬的逻辑来驱动故事。六壬不是门槛，是引擎。

**品牌调性**：不是"算命"，是"平行时空的命运模拟器"。

---

## 二、两条入口（入口层：降低认知负荷，强化代入感）

### 入口一：真实推演 — "平行时空"

用户输入真实的人和事，系统用大六壬排盘演算，生成故事走向。

**"虚实边界"处理**：
- 真实输入容易让用户产生"算命"期待 → UI 文案柔化为"基于现实参数的平行时空推演"
- 入口标题不用"算卦"，用类似"平行推演"或"命运模拟"的措辞
- 适度声明：这是创意工具，不是占卜

**起课仪式感**：
- 系统自动抓取当前时间起课（用户无需手动选时间）
- 如果用户输入了地点，系统提取经纬度
- **定盘动画**：屏幕上短暂闪过一行天文学参数（节气、月将、日缠），然后"啪"地一下定盘
- 动画时长控制在 1-2 秒，制造仪式感但不拖沓

**用户操作流**：
```
输入人物描述 + 事件
        ↓
系统自动提取关键信息，自动获取当前时间/地点
        ↓
【定盘动画】节气、月将、日缠参数闪过 → 定盘
        ↓
大六壬排盘（四课、三传、天将、神煞等）
        ↓
全局故事板（先亮起初传 + 四课）
        ↓
标签点击 + 对话追问 → 逐步解锁中传、末传
```

**举例**：
> "一个40岁的离婚女律师，接到十年前初恋的电话，对方约她见面，说有一份重要的合同要谈。"

系统自动取当前时间（2026年3月），提取人物特征（女、40岁）、事件（约见、合同），起课排盘。

### 入口二：创意起盘 — "命运盲盒"

用户抛一个故事灵感或角色设定，系统用六壬模型推演"这个故事会怎么发展"。

**提供两种起盘形式（用户自选）**：

**形式 A：命运盲抽（塔罗感）**
```
┌─────────────────────────────────────────┐
│                                         │
│    [?]    [?]    [?]    [?]             │
│     牌     牌     牌     牌             │
│    盖着   盖着   盖着   盖着            │
│                                         │
│    点击翻开第一张牌 →                     │
│                                         │
│    [侦探]  [?]    [?]    [?]             │
│     主角   盖着   盖着   盖着            │
│                                         │
│    不满意？点击重洗此牌 ↻                 │
│                                         │
└─────────────────────────────────────────┘
```
- 4 张牌依次翻开，分别对应：主角类型、故事基调、时代背景、关键道具
- 每张牌翻开时有一个翻牌动画
- 用户可以单独重洗任何一张（契合六壬的随机触机感）
- 可选：翻开所有牌后，加一个"注入灵魂"输入框，让用户补充关键细节

**形式 B：造句填空（Mad Libs）**
```
┌─────────────────────────────────────────┐
│                                         │
│  在 「  当代都市  」，一个 「  离婚女律师  」   │
│  卷入了一场 「  黑色幽默  」的事件，       │
│  关键线索是 「  一份十年前的合同  」。      │
│                                         │
│  ↑ 点击高亮词汇即可替换 ↑                 │
│                                         │
│                      [ 推 演 命 运 ]     │
│                                         │
└─────────────────────────────────────────┘
```
- 给出固定句式，高亮词是可替换的
- 点击高亮词弹出选项列表（也可以自定义输入）
- 句式本身会根据用户的选择实时调整语法（如"卷入了"→"邂逅了"→"发现了"）

### 共同引擎：标签点击 + 对话追问

不管从哪个入口进来，最终都进入同一个交互界面——**全局故事板**。

## 三、三层交互模型（核心层：信息分层，焦点引导）

### 第一层：全局故事板（Canvas）— 节点解锁机制

界面主体，视觉化呈现三传结构。**核心原则：不要一次性铺满，要制造悬念。**

**解锁逻辑**：
```
初始状态：只亮起「初传·起因」+ 「四课」（人物初始关系）
    ↓
用户阅读起因，进行几轮标签追问后...
    ↓
出现「推进命运 →」按钮，点击解锁「中传·发展」
    ↓
用户阅读发展，继续追问...
    ↓
再次点击「推进命运 →」，解锁「末传·结局」
```

**故事板布局**：

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│  ┌─ 四课（人物关系快照）─────────────────────────────┐     │
│  │  日干上：女律师（表面）  日支上：初恋（表面）       │     │
│  │  日干下：女律师（内心）  日支下：初恋（真相）       │     │
│  └────────────────────────────────────────────────────┘     │
│                                                          │
│  【初传·起因】    【中传·发展】      【末传·结局】         │
│  ┌──────────┐    ┌──────────┐      ┌──────────┐          │
│  │ ✦ 已解锁  │    │  🔒      │      │  🔒      │          │
│  │          │    │  未解锁   │      │  未解锁   │          │
│  │ 女律师    │    │          │      │          │          │
│  │ 赴约，    │    │  ── ? ──│      │  ── ? ──│          │
│  │ 发现初恋  │    │          │      │          │          │
│  │ 带来了一份│    │          │      │          │          │
│  │ 致命合同。│    │          │      │          │          │
│  │          │    │          │      │          │          │
│  │ ○ 青龙   │    │          │      │          │          │
│  │ ○ 玄武   │    │          │      │          │          │
│  └──────────┘    └──────────┘      └──────────┘          │
│                                                          │
│              [ 推 进 命 运 → ]                            │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**四课与三传的关系**：
- 四课是"现状快照"（当事人和对方各自的表象与真相）— 始终可见
- 三传是"故事弧线"（事态的起承转合）— 逐步解锁
- 用户先通过四课了解人物关系全貌，再沿着三传追踪故事发展

**六壬标签体系**（悬浮在故事段落旁）：
- **天将**（十二神）：代表角色/力量 — 玄武=暗中搅局者，青龙=贵人/机遇，螣蛇=惊变/怪事
- **天盘地盘**：代表时空/五行 — 子水落午位=水火相冲=冲突
- **神煞**：代表特殊剧情 — 驿马=突然变故/出行，天乙=贵人相助

### 第二层：标签点击（聚焦展开）— 人话优先，专业折叠

用户点击故事板上的六壬标签，弹出解释面板。**核心原则：默认只给人话，专业内容折叠收起。**

```
┌──────────────────────────────────────┐
│  青龙（木·吉神）                       │
│                                       │
│  【故事映射】                          │
│  白虎临门：主角遭遇了强力机构的         │
│  暴力介入，一场突如其来的法律诉讼       │
│  让她措手不及。                        │
│                                       │
│  ▸ 盘面解读（展开查看六壬原文）         │
│    ┌────────────────────────────┐     │
│    │ 日干上乘白虎，金克木，     │     │
│    │ 主血光之灾或官方诉讼。     │     │
│    │ 落空亡，事虚而不实...      │     │
│    └────────────────────────────┘     │
│                                       │
│  ──────────────────────────────────  │
│  💬 继续追问...                        │
│  [                                   ]│
└──────────────────────────────────────┘
```

**信息分层**：
1. **第一层（默认展示）**：故事映射 — 纯人话，所有人都能读懂
2. **第二层（折叠/小字）**：盘面解读 — 六壬专业解释，满足考据党的求知欲，但不碍眼

### 第三层：对话追问（开放深钻）— Git 式平行分支

在解释面板底部的输入框中，用户可以：

- **针对当前标签追问**："这个审计师具体是谁？他和女律师父亲是什么关系？"
- **跨标签关联**："玄武说的那个陷阱和青龙说的审计师有关系吗？"
- **主动改变走向**："如果女律师拒绝见初恋，故事会怎么发展？"

**平行分支机制**（最强功能）：

当用户触发"逆天改命"式追问时，系统不是简单回答，而是：
1. 基于新条件重新推演，生成一条新的故事线
2. 在 UI 上可视化呈现分支结构（类似 Git 分支图或《底特律：变人》剧情树）
3. 主线是原三传走向，支线是用户干预后的新走向
4. 用户可以在主线和支线之间切换对比

```
主线（原始推演）
  │
  ├── 女律师赴约 → 合同陷阱 → 反杀
  │
  └─┬─ 支线A："如果她拒绝见面呢？"
    │    初恋直接上门 → 更危险的对抗
    │
    └─┬─ 支线B："如果合同是合法的呢？"
         真正的危机藏在合同的附件里
```

**分支管理**：
- 每条分支都是一个独立的快照，可以随时回切
- 分支可以继续派生子分支（理论上无限层级）
- 用户可以选择"合并"某条支线的元素到主线（类似 Git merge）

## 四、用户权限："反悔权" — 平行卷宗

**支持回到任何一步改条件重新起盘。**

### 命名与包装

- 不叫"快照"或"历史记录"，叫**「平行卷宗」**
- 每个快照自动生成一个 **4 字叙事名**（由 AI 根据盘面内容生成）
  - 例如："夜雨孤舟"、"暗影迷局"、"金蝉脱壳"、"龙虎相争"
  - 而不是显示冷冰冰的时间戳 "14:05:23"

### UI 呈现

```
┌─────────────────────────────────────────────────────────┐
│  平行卷宗                                                │
│                                                         │
│  ◀ [夜雨孤舟] [暗影迷局] [金蝉脱壳] [龙虎相争] ▶  当前  │
│     14:05      14:12      14:18      14:25              │
│                                                         │
│  ← 4个卷宗，横向滑动，点击切换 →                          │
└─────────────────────────────────────────────────────────┘
```

- 横向滑动的卡片列表（类似书签或胶卷）
- 每张卡片显示：AI 生成的 4 字名 + 时间
- 切换时故事板无缝过渡到对应卷宗的完整状态
- 用户可以在任何卷宗的基础上"开新盘"（衍生分支）

### 数据持久化

- 使用 IndexedDB 存储（容量远大于 localStorage）
- 跨会话持久化——关闭浏览器后重新打开，卷宗仍在
- 可选：提供"清空所有卷宗"功能（需要二次确认）

## 五、视觉设计方向

**核心原则**：克制的华丽。高级感来自克制，不是堆砌。

**三个关键词**：简约、高级、品质感

### 风格融合

| 来源 | 应用 |
|------|------|
| **Apple 极简** | 大量留白、清晰的层级关系、功能优先的排版 |
| **包豪斯** | 纯粹几何图形（圆、方、线）作为设计元素，无多余装饰 |
| **Dior 高奢** | 精致细线、克制的金属色点缀、高级字体、材料质感 |

### 配色方案（精确）

| 角色 | 色值 | 说明 |
|------|------|------|
| 背景主色 | `#F9F8F6` | 暖白/米白，有纸张质感 |
| 背景辅助 | `#F2F0ED` | 略深的暖灰，用于卡片/面板 |
| 主文字 | `#1A1A1A` | 近黑，高对比度 |
| 次文字 | `#6B6B6B` | 中灰，用于辅助说明 |
| 强调色（金） | `#C5A059` | 深金/麦穗色，用于六壬符号、关键交互 |
| 辅助色 | `#8B9DAF` | 低饱和蓝灰，用于标签和次要信息 |
| 危险/冲突 | `#C45B5B` | 柔和红，用于白虎等凶神标签 |
| 吉利/机遇 | `#6B9E78` | 柔和绿，用于青龙等吉神标签 |

### 字体方案

| 用途 | 字体 | 说明 |
|------|------|------|
| 大标题 | 思源宋体（Noto Serif SC） | 古典气质，用于产品名、卷宗名等 |
| 正文 | PingFang SC / Inter | 无衬线体，长时间阅读不累眼 |
| 六壬术语 | 思源宋体 | 传统术语用衬线体，区分于正文 |
| 英文/数字 | Inter | 现代感，与中文正文搭配 |

### 六壬符号的视觉化

**核心原则：几何极简，拒绝传统算命刻板印象。**

- 不用云纹、龙纹、八卦图等传统算命视觉元素
- 用**纯几何图形**代表天将和神煞：
  - 每个天将 = 一个极简的几何图标（金色细线描边圆形/方形/菱形）
  - 吉神和凶神用不同的填充/描边样式区分
  - 标签本身就是一个精致的几何线条卡片，没有任何多余装饰
- 视觉上让人联想到**建筑图纸**或**天文星图**，而非算命摊

### 动效原则

- **定盘动画**：天文参数闪过 → 定盘（1-2秒），有仪式感
- **翻牌动画**：盲抽模式翻牌时，卡片翻转效果
- **解锁动画**：中传/末传解锁时，从模糊到清晰的淡入效果
- **分支生长**：Git 式分支出现时，线条生长动画
- **所有动画克制**：不超过 300ms（定盘动画除外），不花哨

## 六、技术方案

### 前端架构

- **框架**：Vite + React + TypeScript
- **样式**：Tailwind CSS（适配包豪斯几何感）
- **状态管理**：Zustand（轻量，适合复杂状态如卷宗/分支管理）
- **动画**：Framer Motion（翻牌、解锁、分支生长等）

### AI 能力

- **API**：Gemini API（故事生成、标签解读、对话追问、卷宗命名）
- **关键设计**：所有 AI 调用**强制 JSON 输出**格式

```json
// 三传故事大纲的返回格式示例
{
  "story_stages": {
    "initial_transmission": {
      "title": "赴约",
      "description": "女律师赴约，发现初恋带来了一份致命的商业合同。",
      "tags": [
        { "type": "deity", "name": "青龙", "element": "木", "nature": "吉" },
        { "type": "deity", "name": "玄武", "element": "水", "nature": "凶" }
      ]
    },
    "middle_transmission": { "..." : "..." },
    "final_transmission": { "..." : "..." }
  },
  "four_courses": {
    "day_stem_upper": { "meaning": "女律师的表面状态", "description": "..." },
    "day_stem_lower": { "meaning": "女律师的真实内心", "description": "..." },
    "day_branch_upper": { "meaning": "初恋的表面", "description": "..." },
    "day_branch_lower": { "meaning": "初恋的隐藏目的", "description": "..." }
  }
}
```

- **Prompt 构造**：前置排盘完成后，将结构化的六壬数据注入 Prompt 模板，要求 Gemini 严格返回 JSON
- **前端解析**：直接 `JSON.parse()` 渲染到对应组件，避免正则解析文本

### 六壬排盘逻辑

- **前端 JavaScript 实现基础排盘算法**（四课三传、天将、神煞）
- 排盘结果序列化为 JSON，注入 AI Prompt
- AI 负责"盘面→故事"的翻译层（这是核心价值，排盘只是计算）

### 数据存储

- **IndexedDB**（通过 Dexie.js 封装）
- 存储：所有卷宗（含盘面数据、故事内容、对话历史、分支结构）
- 跨会话持久化，关闭浏览器不丢失

### 部署

- **纯静态部署**（Vercel / Netlify / GitHub Pages）
- 无需后端服务器
- Gemini API Key 存储在用户本地（首次使用时输入，加密存入 IndexedDB）
